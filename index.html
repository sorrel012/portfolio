📁 ERD
💻 화면 설계
🔨 기능
1️⃣ 주식 시세 크롤링
네이버 증권에서 주식의 시세를 얻어와서 데이터베이스에 저장하는 기능입니다.
jSoup 라이브러리를 이용하여 시세 정보를 크롤링하여 list에 추가합니다. 그 후 반복문의 index를 적절히 활용하여 list의 요소를 종목별로 나누어 DB의 코스피, 코스닥 테이블 각 행에 저장합니다.
코스피와 코스닥 테이블의 데이터를 각각 상승률, 하락률 순위대로 정렬하여 view를 생성합니다. 상승률/하락률 Top 10의 시세정보를 조회하는 기능을 view를 이용하여 간단하게 구현할 수 있습니다.
2️⃣ 종목 검색
종목명을 입력하여 원하는 종목의 시세를 검색하는 기능입니다. 회원의 경우 검색한 종목을 관심 종목으로 등록하거나 매수할 수 있습니다.
검색한 종목을 관심 종목으로 등록하거나 매수할 수 있는 메소드입니다. 비회원일 경우 NullPointerException을 발생시킵니다. 해당 예외는 userAdditionalInput()에서 처리하여 로그인하지 않은 경우 해당 기능을 이용할 수 없음을 안내합니다.
3️⃣ 개인정보 조회 및 수정
로그인한 사용자가 자신의 개인정보를 조회하고 수정할 수 있는 기능입니다. 비밀번호, 이메일, 연락처를 수정할 수 있으며 회원 탈퇴도 이루어집니다.
사용자의 이메일 변경을 처리하는 메소드입니다. 먼저 현재 이메일 주소와 새로운 이메일 주소가 동일한지 검증한 후, 동일할 경우 안내 메시지를 출력 후 다시 입력을 받습니다. 동일하지 않을 경우 이메일 형식이 올바른지검증 하여 올바를 경우 이메일 주소를 변경합니다.
4️⃣ 랭킹 조회
사용자 자산 랭킹을 조회하는 기능입니다. 자신의 등수를 알 수 있고, 총 자산이 가장 많은 사용자 10명의 자산 정보를 조회할 수 있습니다.
전체 사용자의 자산을 정렬하는 메소드입니다. 먼저 사용자의 고유 번호가 key, 총 자산이 value인 HashMap에서 key와 value를 Entry 형식으로 변환합니다. 그 후 총 자산인 value를 기준으로 내림차순 정렬하고, 만약 총 자산이 같을 경우 고유 번호인 key를 기준으로 오름차순 정렬합니다.

<팀 프로젝트>

처음으로 진행하는 팀프로젝트로, 프로젝트 초기 단계에서 많은 이야기룰 나누었습니다. 특히 설계 과정에서 활발한 상호작용을 통해 기능을 세부적으로 분배하여 공통된 메소드를 추출했습니다. 이를 통해 같은 작업을 반복하는 일을 줄여 효율성을 높일 수 있었습니다. 기능을 구현할 때는 처음에 branch를 나누지 않고 main branch에서 모두 작업하다 보니 계속 충돌이 발생하였습니다. 이를 해결하기 위해 각자 branch를 생성하고, 기능 구현을 마친 후 팀원들과 공유한 후 병합하는 방법을 선택했더니 오류를 줄일 수 있었습니다. 이처럼 구현 중에도 실시간으로 작업하며 오류 없이 진행하기 위해서 끊임없이 이야기를 나누어야 했습니다. 소통이라는 팀 단위 작업에서 가장 중요한 것은 소통이라는 것을 깨달을 수 있었던 프로젝트였습니다.

<개인 프로젝트>

팀 프로젝트를 하면서 느꼈던 불편한 점을 개선하기 위해 개인 프로젝트를 시작했습니다. 팀 프로젝트 당시에는 파일 입출력을 통해 데이터를 관리했기 때문에 데이터의 특정 부분에만 접근할 수 없어서 전체 데이터를 다 읽어온 후 조작하고 다시 전체를 저장해야 했습니다. 데이터베이스에 대해 학습하며 이러한 불편함을 해결할 수 있을 것 같다는 생각이 들어 파일입출력 구조에서 데이터베이스로 전환하는 개인 프로젝트를 추가로 진행하게 되었습니다.

개인 프로젝트를 진행할 때는 기존 팀 프로젝트에서 다른 팀원들이 구현한 코드도 수정해야 했기 때문에 많은 이해와 노력이 필요했습니다. 그렇지만 프로젝트를 마치고 나니 파일 입출력보다 훨씬 더 간단하게, 효율적으로 데이터를 관리할 수 있었습니다. 직접 겪었던 불편함을 해결하기 위해 데이터베이스를 활용하면서 데이터베이스의 필요성을 확실히 느낄 수 있었습니다.

먼저 개념적 모델링을 통해 교육 센터 운영에 필요한 개체, 관계, 속성을 정리하였습니다. 정리한 개체, 관계, 속성을 바탕으로 논리적 데이터 모델링을 진행하였습니다. 정규화 과정을 거쳐 논리적 데이터 모델을 설계한 후 최종적으로 물리적 모델링을 거쳐 교육 관리 시스템 운영을 위한 관계형 데이터베이스를 구현하였습니다.

교육 관리 시스템

📁 ERD
🔨 기능
1️⃣ 교사 정보 관리
교사의 개인정보를 입력, 조회, 수정, 삭제하는 기능입니다.
교사의 개인정보를 입력하는 저장 함수입니다. 오류가 발생하지 않고 정상적으로 입력되었을 경우 저장 함수의 반환값인 '입력되었습니다.'를 출력합니다. 만약 오류가 발생하여 정상적으로 처리되지 않았을 경우 호출할 때 예외처리를 통해 '입력에 실패하였습니다.'를 출력합니다.
2️⃣ 개설 과정 정보 관리
개설된 과정명, 과정 진행 날짜, 강의실, 과목 등록 여부, 교육생 등록 인원을 입력, 조회, 수정, 삭제하는 기능입니다.
개설 과정 정보를 삭제, 수정하는 프로시저입니다. 특정 과정을 삭제할 때는 삭제할 과정 번호를 매개변수로 입력하여 과정 관련 테이블에서 해당 과정을 삭제합니다. 특정 과정 정보를 수정할 때는 수정 내용과 과정 번호를 매개변수로 입력하여 원하는 수정 내용 관련 테이블에서 해당 정보를 수정합니다.
3️⃣ 특정 과정 정보 조회
과정명, 과정 진행 날짜, 강의 교사, 강의 과목 등 개설된 과정에 대한 전반적인 정보를 조회하는 기능입니다.
특정 과정 번호를 입력하여 해당 과정에 관한 정보를 조회할 수 있는 쿼리입니다. 많은 테이블을 join하여 데이터를 탐색해야 하는 기능이기 때문에 사전에 view를 생성하였습니다. 이를 통해 가독성을 높이고 이해하기 쉬운 쿼리를 구현했습니다.
4️⃣ 과정 수료 시 수료 날짜 지정
과정 종료 시 교육생의 수료 상태와 수료 날짜를 변경하는 기능입니다.
과정 종료 시 종료된 과정의 번호를 매개변수로 받아 중도탈락을 제외하고 해당 과정에 참여한 모든 학생들의 수료 상태를 '수료'로 지정하는 프로시저입니다. 수료 상태가 '수료'로 변경되면, 트리거에 의해 수료 날짜가 과정 종료 날짜로 자동 지정됩니다.
🧾 소감

이번 프로젝트의 핵심이 되는 것은 ERD 모델이었습니다. 프로젝트 초기 회의 때 팀원들과 모델링을 꼼꼼히 수행한 후 업무 쿼리를 짜기로 이야기를 나누었습니다. 여러 차례의 정규화를 통해 데이터의 중복이 발생하지 않도록 테이블을 분리하였습니다. 이를 바탕으로 수월하게 데이터를 정의할 수 있었습니다.

그러나 문제는 데이터를 조작할 때 발생했습니다. 정규화 과정은 거쳤지만, 반정규화 과정을 제대로 거치지 않은 것이었습니다. 정규화 당시에는 join을 하면 데이터를 조작하는 것이 쉬울 것이라고 생각하여 반정규화를 간과하고 물리적 모델링을 수행했던 것입니다. 본래 계획대로 join을 통해서 데이터를 조작하는 것이 가능하기는 했지만, 많은 테이블을 join하게 되면서 가독성이 떨어지고 쿼리를 보고 업무를 이해하기가 어려웠습니다. view를 생성하여 어느 정도의 성능 및 가독성을 높일 수 있었지만, 그 전에 테이블을 적절히 통합했다면 더욱 효과적이었을 것입니다. 데이터베이스를 설계할 때는 정규화뿐만 아니라 적절한 반정규화를 통해 효율성을 향상시킬 필요가 있다는 것을 깨달았습니다.
